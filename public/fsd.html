High Level Process & Workflows

The AI Essay Preparedness Grader is a microservices-based system designed to evaluate a student's readiness for a specific college major through an interactive AI-driven interview session. The following workflows describe the core processes of the system.

1. Complete Grading Session Flow

This is the primary user journey, from login to receiving the final report.

  1. Authentication: User logs in via Frontend. API Gateway routes to Auth Service, which validates credentials against the Database and returns a JWT.
  2. Session Initiation: User requests to start a grading session. Session Service creates a new session record in the Database.
  3. Real-time Connection: Frontend connects to the WebSocket Service using the session ID.
  4. AI Conversation Loop:
    • Question: Grading Session Service requests a question from the AI Service (Grading Worker). AI Service calls Google Gemini API. The question is saved to the DB and sent to the user via WebSocket.
    • Answer: User submits an answer. It is sent via WebSocket to the Grading Session Service.
    • Analysis: The answer is saved and sent to the AI Service for analysis. Google Gemini evaluates the answer and assigns a score.
    • Scoring: The score is updated in the Database. If the threshold is met or max questions reached, the loop ends.
  5. Completion: The session is marked as completed.
  6. Reporting: AI Service generates a comprehensive final report. Result Service saves this report. The user receives the final result via the Frontend.

2. User Registration Flow

  1. User submits registration form.
  2. Auth Service creates a user in Firebase Authentication.
  3. Upon success, Auth Service creates a corresponding user profile in the local PostgreSQL Database.
  4. Registration success response is returned to the user.

3. Real-time Communication Flow

  1. Connection: Frontend connects to WebSocket Service with a JWT.
  2. Validation: WebSocket Service validates the session with Grading Session Service.
  3. Exchange:
    • AI -> User: Questions are broadcasted to the specific session room.
    • User -> AI: Answers are emitted to the server, processed, and acknowledged.
  4. Disconnection: On session end or network loss, the connection is closed. Reconnection logic handles state recovery.

4. AI Service Integration Flow

  1. Question Generation: AI Service checks Cache (Redis) for similar questions. If miss, calls Google Gemini with a prompt template.
  2. Answer Analysis: User answer is analyzed against the question context using Gemini. Result includes a numerical score and qualitative feedback.
  3. Final Report: Aggregates the entire conversation history and requests a summary analysis from Gemini (Strengths, Weaknesses, Recommendations).

5. Error Handling & Recovery

  • AI Service Error: If Gemini fails, the system retries. If it persists, the session is marked as failed or degraded, and the user is notified.
  • Database Error: Connection failures trigger retry logic. Critical failures notify the user of service degradation.
  • WebSocket Disconnection: Frontend attempts reconnection. Missed messages are retrieved from the message history in the Database.

6. Session Timeout & Cleanup

  • Scheduler: Periodically checks for active sessions that have exceeded their expires_at time.
  • Cleanup: Expired sessions are marked as expired. WebSocket connections are forced closed. Partial results may be generated if sufficient data exists.

Relationships

  • Users 1:N Grading Sessions: A user can have multiple grading sessions.
  • Grading Sessions 1:N Messages: A session consists of multiple messages.
  • Grading Sessions 1:1 Grading Results: A session produces exactly one final result.

API Specification & Features

The system exposes a RESTful API via the API Gateway. All endpoints (except Auth) require a Bearer <jwt_token> header.

1. Auth Service (/auth/*)

Handles user identity and profile management.

  • POST /auth/register: Register a new user.
  • POST /auth/login: Login and retrieve JWT.
  • POST /auth/refresh: Refresh an expired access token.
  • POST /auth/logout: Invalidate the current session.
  • GET /auth/me: Retrieve current user's profile.
  • PATCH /auth/profile: Update user profile details.
  • POST /auth/forgot-password: Initiate password reset flow.
  • POST /auth/reset-password: Complete password reset.
  • DELETE /auth/user: Delete user account.

2. Session Service (/grading-sessions/*)

Manages the grading session lifecycle.

  • POST /grading-sessions: Create/Start a new grading session.
    • Payload: target_major, max_questions, session_duration_minutes.
  • GET /grading-sessions/{session_id}: Get details of a specific session.
  • GET /grading-sessions: List user's sessions (supports pagination & filtering).
  • POST /grading-sessions/{session_id}/complete: Manually complete a session.
  • GET /grading-sessions/{session_id}/messages: Retrieve chat history for a session.
  • POST /grading-sessions/{session_id}/messages: Send a message (fallback for WebSocket).

3. Result Service (/grading-results/*)

Retrieves final grading outcomes.

  • GET /grading-results/{session_id}: Get the result for a specific session.
    • Response: Includes final_score, readiness_level, and the full analysis_report.
  • GET /grading-results: List user's past results.

4. WebSocket Service (Socket.io)

Handles real-time interaction during the session.

  • Namespace: / (Root)
  • Events:
    • join_session: Client joins a session room.
    • question: Server sends a new question to Client.
    • answer: Client sends an answer to Server.
    • score_update: Server sends real-time score feedback.
    • session_completed: Server notifies that the session has ended.
    • error: Server sends error details (e.g., session expired).

5. Error Handling

Standardized error response format:

{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "Human readable message",
    "details": { ... }
  }
}

Common Error Codes:

  • UNAUTHORIZED (401): Missing/Invalid Token.
  • FORBIDDEN (403): Access denied.
  • NOT_FOUND (404): Resource not found.
  • VALIDATION_ERROR (400): Bad request data.
  • SESSION_EXPIRED (410): Session is no longer active.
  • RATE_LIMIT_EXCEEDED (429): Too many requests.
